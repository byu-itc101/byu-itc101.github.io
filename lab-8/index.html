Overview

This mini-game introduces basic Linux firewall administration using UFW and walks through a realistic security-first server configuration.

What is a Firewall?

A firewall is a software or hardware security device that monitors and controls incoming and outgoing network traffic based on a set of security rules. You can think of a firewall like a security gate:

  • It looks at each packet of data entering or leaving your computer.
  • It checks those packets against predefined rules.
  • It either allows or blocks the packet.

By default, most servers allow all traffic, which can expose them to threats. Firewalls help you reduce your attack surface, prevent unauthorized access, and monitor suspicious traffic.

How Do Firewalls Work?

A firewall intercepts data at the network level. Packets are small chunks of data that computers send over the network and include headers with details such as:

  • Source IP address
  • Destination IP address
  • Source and destination ports, such as 22 for SSH and 80 for HTTP
  • Protocol, such as TCP, UDP, or ICMP

When a packet arrives:

  1. The firewall looks at its contents.
  2. Starting with the first rule, it compares the packet against your rule list from top to bottom.
  3. If a match is found, such as a rule allowing SSH on TCP port 22, the packet is accepted.
  4. If no allow match exists and you have a default deny, or if the packet matches a deny rule, the packet is dropped.

What Is a Default Deny Rule?

A default deny rule, also called an implicit deny, blocks any traffic not explicitly allowed by earlier rules. It is typically placed at the bottom of the rule set and is considered a best practice because it ensures that only known and intended traffic is allowed.

This is different from an explicit deny rule, which you place somewhere in your rules list to proactively block certain traffic before it reaches a later allow rule.

What is UFW?

UFW, or Uncomplicated Firewall, is a user-friendly frontend for managing iptables on Linux. Its primary purpose is to help you control which incoming and outgoing network connections are allowed on your computer without needing to write complicated firewall rules manually.

Why use UFW?

  • Easy setup for basic security
  • Simplifies complex firewall rules
  • Provides numbered lists so you can insert or delete rules at specific positions
  • Allows filtering by service, port, protocol, and even source/destination IP

What is iptables?

iptables is a powerful command-line utility that allows system administrators to define the rules that control network traffic. It is part of the Linux kernel and interacts with the netfilter framework, which processes packets as they come into or leave the system. With iptables, you can allow or block packets, forward them to different networks, log traffic, and filter based on IP addresses, ports, protocols, and more.

Although iptables is very capable, its raw commands can be quite complex. UFW exists to simplify its use.

What is CIDR Notation?

CIDR stands for Classless Inter-Domain Routing. It is used to specify IP address ranges in a compact format.

172.16.72.0/24 means:

  • IP range: 172.16.72.0 through 172.16.72.255
  • /24 means the first 24 bits are fixed as the network portion and the remaining 8 bits are used for host addresses
  • This creates a subnet with 256 total IP addresses, 254 of them usable for hosts

Objectives

Your goal is to secure the server so that only the services that are absolutely required are accessible to the right people. Specifically, you want to allow public visitors to access the web service over HTTP and HTTPS while ensuring that only trusted server administrators can log in via SSH.

In a real-world setup:

  • The public only needs access to your web application, typically on port 80 for HTTP and port 443 for HTTPS.
  • Server administrators need SSH access, usually on port 22, and that access should be restricted to specific IP addresses or networks.
  • Other ports and services, such as databases, administrative panels, or debugging tools, should not be publicly accessible at all.

Leaving ports open that do not need to be open is a serious security risk. Every open port is a potential entry point for attackers.

This lab also illustrates defense in depth. Even if an attacker somehow obtains an SSH password, a properly configured firewall can still prevent them from reaching the service at all.

By the end of this mini-game, you will:

  • Configure UFW with a default deny policy for incoming traffic
  • Allow HTTP and HTTPS so the public can view your website
  • Allow SSH, restricted so only administrators can connect
  • Practice inserting rules at specific positions
  • Practice blocking individual IPs
  • Practice specifying protocols such as TCP or UDP

Steps

Step 1: Check UFW Status

Before making any changes to your firewall rules, it is important to understand the current status of UFW.

Type this command and press Enter:

sudo ufw status verbose

sudo stands for "super user do." It allows you to run commands with elevated privileges, which are often required for administrative tasks like configuring the firewall.

When you type a command with sudo, you might be prompted to enter your password. This is a safety feature to ensure only authorized users can make critical system changes.

This command displays UFW's current status and firewall rules. By default, UFW is often inactive on a machine. If enabled, you might see output like this: 

Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), disabled (routed)
New profiles: skip

Status:

  • active means UFW is enabled and enforcing its rules
  • inactive means UFW is installed but not running, so the rules are not currently being applied

Logging:

Logging shows whether UFW is recording connection attempts. Common values include off, low, medium, and high.

Default policies:

  • incoming: the action UFW takes on incoming traffic that does not match an allow rule
  • outgoing: the action UFW takes on outgoing traffic that does not match a specific rule
  • routed: the action for routed packets, typically disabled unless the server is acting as a router

Other useful ufw status options:

  • sudo ufw status for a quick summary of rules and the enabled or disabled state
  • sudo ufw status numbered to display rules with line numbers so you can delete or insert them at specific positions
  • sudo ufw status verbose for detailed output including default policies, logging, and new profiles

Step 2: Enable UFW and Set Default Rules

Your company's security policy requires that:

  • All incoming connections are denied by default unless explicitly permitted
  • All outgoing connections are allowed by default so the server can communicate externally

Why set default rules before enabling UFW?

It is important to configure these policies before turning the firewall on because enabling UFW first may block legitimate traffic unexpectedly, potentially locking you out of the server or disrupting services.

Why are these default policies considered best practice?

  • Deny all incoming by default: this follows the principle of default deny, meaning anything not explicitly allowed is blocked
  • Allow all outgoing by default: this allows the server to initiate updates, API calls, and other normal outbound traffic without unnecessary restrictions

Run the following commands:

sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow ssh
sudo ufw enable
sudo ufw status verbose

The first two commands set the default policies for incoming and outgoing traffic. The third command allows SSH traffic into the server. If you omit this rule, you can lock yourself out of the server.

The fourth command activates UFW. You may see the warning Command may disrupt existing ssh connections. Type Y only if you have already run sudo ufw allow ssh.

The last command checks the firewall status and confirms that the defaults and active state are as expected.

What to expect:

  • Status: active indicating UFW is enabled
  • Default policies set to deny incoming and allow outgoing
  • Logging settings and other details

Step 3: Allow Web and SSH Services

To make your server functional and accessible, you need to explicitly allow certain incoming traffic through the firewall. For most web servers, these include:

  • SSH for remote administrative access, usually on port 22
  • HTTP on port 80
  • HTTPS on port 443

Note about SSH:

In many Linux distributions, UFW includes predefined application profiles and may have SSH allowed by default to prevent lockouts. However, you have already allowed incoming SSH in the previous step. You should explicitly allow SSH rather than relying on defaults so your access remains intact even if the defaults change or are misconfigured.

Allow these services with the following commands:

sudo ufw allow http
sudo ufw allow https

Different ways to allow traffic with UFW:

  1. By service name, such as ssh, http, or https, which refers to predefined profiles
  2. By port number, useful when a service uses a non-standard port
  3. By port and protocol, such as TCP or UDP

Examples:

  • sudo ufw allow ssh
  • sudo ufw allow 22
  • sudo ufw allow 80
  • sudo ufw allow 443
  • sudo ufw allow 22/tcp
  • sudo ufw allow 80/tcp
  • sudo ufw allow 443/tcp

Verify the rules:

After allowing these services, check the current rule list:

sudo ufw status numbered

This will show a numbered list of active firewall rules.

Step 4: Allow Traffic by IP Address

The rule you created earlier allows SSH connections from any IP address. To improve security, you should restrict SSH access to trusted management devices. In this lab, the trusted subnet is 172.16.72.0/24.

SSH uses the TCP protocol over port 22. Use the following command format to allow only the trusted subnet to access the VM by SSH: 

sudo ufw allow proto <tcp|udp> from <IP Address>/<CIDR> to any port <Port Number>

For this lab, that becomes:

sudo ufw allow proto tcp from 172.16.72.0/24 to any port 22

Note: Whenever you see something surrounded by <>, replace the contents and the angle brackets with the correct value. The | symbol means "or."

UFW processes rules top-down in order. If a previous rule already allows SSH from anywhere, it will override the new restricted rule unless you remove or reorder it.

Check the current rule list with:

sudo ufw status numbered

Look for an existing rule similar to one of these: 

[1] SSH                        ALLOW IN    Anywhere
[2] 22                         ALLOW IN    Anywhere

Your rules may look different, but look for entries with SSH or port 22. Take note of the rule number for any SSH rule that allows access from any IP. You will also need to remove the corresponding (v6) rule if it appears.

To prevent all IPs from accessing the VM via SSH, delete the overly permissive rule using its rule number: 

sudo ufw delete <Rule Number>

For example, if the rule was number 2:

sudo ufw delete 2

Check that only your intended SSH access rule remains:

sudo ufw status numbered

You should see a rule like:

[3] 22                         ALLOW IN    172.16.72.0/24

Remember that after you delete a rule, the rules below it will be renumbered.

Step 5: Blocking Malicious Traffic from a Specific IP Address

The network team has identified malicious activity coming from the IP address 172.16.74.2. To protect your system, you need to block all traffic from this IP.

Use the following command format:

sudo ufw insert <position> deny proto <tcp|udp|any> from <IP address> to any

Position is the number where the rule will be inserted. Position 1 would place the rule first, and position 2 would insert it after the current first rule.

To block all protocols from 172.16.74.2, make sure the deny rule is evaluated before other allow rules.

Verify the rule:

List all current rules with their numbers:

sudo ufw status numbered

Ensure that the deny rule is listed above any allow rules that might otherwise permit traffic from this IP. UFW processes rules top-down, so the position of the deny rule is critical.

Final Check

At this point, your firewall configuration should meet the following criteria:

  • UFW is enabled
  • The default policy denies all incoming traffic and allows all outgoing traffic
  • Known malicious IPs such as 172.16.74.2 are blocked
  • Only trusted services are allowed
  • HTTP on port 80
  • HTTPS on port 443
  • SSH on port 22, restricted to the management subnet 172.16.72.0/24

You can confirm these rules are in place by running:

sudo ufw status numbered
